1. Bug Summary

Regression: All quiz paths use identical linear flow

The nextStep() function in the quiz JavaScript has no conditional branching. Regardless of what the user selects in Step 1, the quiz always progresses through the same 5 steps:

Step 1 (selection) → Step 2 (situation) → Step 3 (finances) → Step 4 (contact) → Step 5 (borrowing result)

This means:

— "Build" should follow the same path as Purchase (borrowing capacity calculator) — this is correct.
— "Commercial or business loan" goes through residential loan questions — HG has limited commercial capacity and is currently not accepting new clients.
— "Refinance" asks "Are you a first home owner?" and "How much is your deposit?" which don't apply.
— "I'm just exploring" forces the same full borrowing calculator, which is too aggressive for explorers.

Root Cause

The nextStep() function simply increments currentStep++ without reading the value of the looking radio input:

function nextStep() {
  // BUG: No branching based on answers.looking
  if (currentStep === totalSteps - 1) {
    const data = calculateLoanCapacity(answers);
    borrow.textContent = data.maxPropertyValue;
    maxBorrow.textContent = data.improvedPropertyValue;
    submitForm();
    return;
  }
  if (currentStep < totalSteps) {
    stepElements[currentStep - 1].classList.remove('active');
    currentStep++;
    stepElements[currentStep - 1].classList.add('active');
    updateButton();
    disableMainButton();
  }
}

There is no code that checks answers.looking (or the radio value for name="looking") to branch the flow.

Edge Cases & Traps for Developers

The following issues were found by inspecting the live JS. These will break your implementation if you don't account for them.

1. disableMainButton() hardcodes step numbers

The button-enable logic checks specific step numbers:

if (currentStep === 1 && !answers.looking) { disabled }
if (currentStep === 2 && (!answers.maritalStatus || ...)) { disabled }
if (currentStep === 3 && (!answers.income || !answers.deposit)) { disabled }
if (currentStep === 4 && (!formData.name || ...)) { disabled }

Problem: If you add new steps (refinance steps, exploring steps), the step numbers won't match. Step 2 for refinance is "interest rate", not "marital status". You need to refactor this to check which fields the current step requires, not hardcoded step numbers.

2. totalSteps = 5 is hardcoded

const totalSteps = 5 controls the progress indicator and the "Submit" button logic. Refinance has 6 steps. Exploring has 2–4. You must make this dynamic per flow.

3. submitToContactForm7() already sends refinance fields — as dashes

The existing submit function sends:

r.append("mortgage-rate", "-");
r.append("still-owe", "-");
r.append("loan-term", "-");
r.append("important", "-");

This is significant: CF7 form #72896 already has fields for refinance data (mortgage-rate, still-owe, loan-term, important) — they're just hardcoded to "-". For the refinance flow, populate these with actual answers instead of adding new fields. This means the CF7 form and HubSpot mapping may already be set up.

4. updateButton() uses totalSteps for Submit vs Continue

if (currentStep < totalSteps - 1) { "Continue" }
else { "Submit" }

With dynamic step counts, this needs to check the current flow's total, not the global totalSteps.

5. answers and formAnswers are separate objects

answers = { looking, maritalStatus, isFirstOwner, state, dependants, income, deposit }
formAnswers = { name, email, phone }

answers feeds the calculator. formAnswers is what gets submitted. For refinance, you need to add interestRate, loanAmount, loanTerm, refiPriority to the submission — but map them to the existing CF7 field names (mortgage-rate, still-owe, loan-term, important).

6. Event listeners are bound by element ID — new fields need new listeners

The existing code binds addEventListener to #income, #deposit, #state, #name, #email, #phone by ID. New refinance fields need their own listeners to update the answers object and call disableMainButton().

7. No analytics tracking exists

The quiz has zero GTM dataLayer pushes, zero Google Analytics events, zero Facebook pixel events. This means there's no tracking of which quiz path users take, where they drop off, or conversion rates per path. Consider adding dataLayer.push events when branching is implemented.

8. No OTP/phone verification on the homepage quiz

Despite the spec mentioning OTP in some flows, the homepage quiz has no SMS verification. The phone field is a plain text input with client-side AU format validation only. The OTP system exists on a separate CRS-wrapped form (Flow 5 in the system docs) but is NOT part of this quiz.

9. stepElements = document.querySelectorAll(".step") runs once at load

This selector captures all .step elements when the page loads. If you add new step divs dynamically (e.g., refinance steps injected via JS), they won't be in this NodeList. Either add all step HTML at build time, or re-query after adding steps.

10. Progress indicator "/ 5" is hardcoded in HTML

The progress indicator is:

<div class="progress_indicator">
  <span id="current_step">1</span> / 5
</div>

The JS updates #current_step text, but the / 5 is static HTML. To make it dynamic, you need to also update or wrap the total in a <span> so JS can change it per flow.

11. Steps have different content heights — no animation

Step heights vary: Step 1 = 376px, Step 2 = 531px, Step 3 = 492px, Step 4 = 486px, Step 5 = 406px. The quiz card has no fixed height, no transition, no animation. When steps switch, the card height jumps instantly. New refinance/exploring steps will have different heights again. Consider adding transition: height 0.3s or min-height to avoid layout jumps.

12. Steps use display: none, not .active class for visibility

Hidden steps have display: none (not just missing the .active class). The JS removes .active from the current step and adds it to the next. Verify that your CSS ties .active to display: block and the default .step is display: none. New step divs need the same CSS rules.

Why Two Forks?

Opt-in borrowing power is the key insight: forcing financials on someone who said "just exploring" is too aggressive and will cause drop-off. But offering it as a value exchange ("see how much you could borrow in 30 seconds") converts curious explorers into warm leads with full data — without the pressure of contact details or phone verification.

Stage-based redirects catch users who selected "exploring" but are actually further along. Rather than giving them a watered-down experience, funnel them into the full Purchase or Refinance flows where they'll get the most value.

Data Passthrough

Explorers who opt into the borrowing power check collect the same fields as the Purchase situation + finances steps (maritalStatus, isFirstOwner, state, dependants, income, deposit). This data feeds directly into the existing calculateLoanCapacity() function. The form submission includes looking=exploring and exploreStage so the CRM can distinguish these leads from direct purchase enquiries.

9. Implementation Notes

Core Change: Branching in nextStep()

The primary fix is modifying nextStep() to read answers.looking after Step 1 and route to the correct flow. Here's the suggested pseudocode:

// Suggested refactored nextStep()
function nextStep() {
  const looking = answers.looking;

  // After Step 1: branch based on selection
  if (currentStep === 1) {
    switch (looking) {
      case 'purchase':
        goToStep('purchase_situation'); // existing step_2
        break;
      case 'build':
        goToStep('purchase_situation'); // Same flow as purchase
        break;
      case 'commercial':
        showDeadEndScreen('commercial');
        return;
      case 'refinance':
        goToStep('refi_situation');
        break;
      case 'exploring':
        goToStep('explore_stage'); // Step 2: What stage?
        break;
    }
    updateProgressIndicator();
    return;
  }

  // Exploring Step 2: branch based on stage
  if (looking === 'exploring' && currentStep === 'explore_stage') {
    const stage = answers.exploreStage;
    if (stage === 'ready_to_buy') {
      answers.looking = 'purchase';
      goToStep('purchase_situation');
      return;
    }
    if (stage === 'curious_refi') {
      answers.looking = 'refinance';
      goToStep('refi_situation');
      return;
    }
    // Early-stage: show borrowing power fork
    goToStep('explore_borrowing_fork');
    return;
  }

  // Exploring borrowing power fork
  if (looking === 'exploring' && currentStep === 'explore_borrowing_fork') {
    if (answers.wantsBorrowingPower) {
      goToStep('explore_situation'); // collect household + finances
    } else {
      goToStep('explore_resources'); // skip to resources
    }
    return;
  }

Note on Step IDs in Pseudocode

The pseudocode above uses descriptive string IDs like 'purchase_situation' and 'refi_situation' for clarity. The actual DOM uses numeric classes (.step_1 through .step_5). When implementing, you'll need to either:

Option A: Add new step divs with new classes (.step_refi_2, .step_explore_1, etc.) and map them in your flow arrays.

Option B: Re-use the existing .step_2 through .step_5 by swapping their inner HTML based on the active flow (lighter DOM, but more complex JS).

Option A is recommended — it's cleaner and avoids state bugs from HTML swapping.
