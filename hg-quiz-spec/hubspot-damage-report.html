• Commission rate of 0.6% used (industry range: 0.5–0.7%). Using 0.7% would increase the figure by 17%.

• Does not include the 80+ additional leads Jayden reported as "completely missed" from the webhook disconnect (reported in a separate URGENT email thread). If these are distinct from the populations above, the total damage is higher.

• The 6 settled deals from contacts with corrupted quiz data averaged $794,433 per deal — higher than the $655,252 used in this calculation.

What's not counted

• Reputational damage from "Dear Mdaniels.Mmd@Gmail.Com" broker emails. Customer trust erosion. Opportunity cost of Jayden's 5+ hours of investigation. The 268 contacts with no form attribution (impossible to assign to the right broker/pipeline). Trail commission beyond year 3.

Issue 1: Email Address Stored as First Name (90 contacts)

CRS Implementation Error: CF7 field mapping put email in firstname

90 contacts in HubSpot have their email address stored in the firstname field. This means broker notifications, email personalisation, and CRM searches all show an email address instead of a name.

Root cause: CRS mapped the CF7 email field (your-email or equivalent) to the HubSpot firstname property instead of the email property in the Collected Form configuration.

Date range: Zero email-as-name contacts in the 4 months immediately before CRS deployed (Jun–Sep 2025). The bug started in October 2025 — the exact month CRS deployed their quiz forms — with 8 contacts. It then escalated to 28/month by February 2026. A rate of 28/month is not users accidentally typing their email into the name field — it is a systematic field mapping defect where the form code itself puts the email value into the firstname property on every submission from CRS-built forms. 81 of the 90 affected contacts (90%) were created after CRS deployed.

Issue 5: 268 Recent Contacts With No Form Attribution

Contacts entering HubSpot with no record of which form captured them

Issue 6: WordPress Site — Frontend Implementation Failures

Playwright scan of the live WordPress site on 2 April 2026 found additional CRS implementation issues:

6a. Homepage quiz JS has console.log in production

The submitToContactForm7() function includes console.log(answers) which fires on every form submission. This leaks quiz answer data to the browser console in production. Left in from development/debugging.

6b. Quiz answers object is never included in form submission

The answers object ({ looking, maritalStatus, isFirstOwner, state, dependants, income, deposit }) is populated from the quiz but never appended to the FormData sent to CF7. Only formAnswers = { name, email, phone } is submitted. The entire quiz — situation questions, finances, borrowing intent — is collected from the user and then thrown away.

6c. /lower-your-rate/ quiz JS does not populate hidden fields

The refinance quiz at /lower-your-rate/ renders questions dynamically from a JSON config, but the submit handler does not write quiz answers to hidden fields before form submission. Confirmed: populatesHiddenFields: false.

6d. Contact page loads 13 CF7 forms

The /contact/ page has 13 CF7 form instances loaded in the DOM — 1 actual contact form (#11830), 3× #14462, 1× #14422, 1× #14296, 3 popup forms (#14061, #14060, #14059), and 4× popup quiz #11832. Massive DOM bloat from the global template.

6e. /mortgage-calculator/ page has no quiz at all

Verified: hasQuizCard: false, hasNextStep: false. The quiz only exists on the homepage (/). The /mortgage-calculator/ page loads 12 CF7 forms from the template but has no quiz card or calculator button.

6f. 403 error on homepage resource load

Console error on homepage: "Failed to load resource: the server responded with a status of 403". A resource is being blocked on the CRS-modified page.
