1. Опис помилки

Регресія: всі шляхи квізу використовують однаковий лінійний потік

Функція nextStep() в JavaScript квізу не має умовного розгалуження. Незалежно від того, що користувач обирає на кроці 1, квіз завжди проходить через однакові 5 кроків:

Крок 1 (вибір) → Крок 2 (ситуація) → Крок 3 (фінанси) → Крок 4 (контакти) → Крок 5 (результат)

Це означає:

— "Будівництво" повинно слідувати тому ж шляху, що й Купівля (калькулятор кредитоспроможності) — це правильно.
— "Комерційний або бізнес-кредит" проходить через питання для житлових кредитів — HG має обмежену пропускну здатність для комерційних кредитів і наразі не приймає нових клієнтів.
— "Рефінансування" запитує "Чи ви перший покупець житла?" та "Який ваш депозит?" — це не стосується рефінансування.
— "Просто дивлюсь" змушує проходити повний калькулятор кредитоспроможності, що занадто агресивно для тих, хто лише досліджує.

Першопричина

Функція nextStep() просто збільшує currentStep++ без зчитування значення radio-кнопки looking:

function nextStep() {
  // БАГ: Немає розгалуження на основі answers.looking
  if (currentStep === totalSteps - 1) {
    const data = calculateLoanCapacity(answers);
    borrow.textContent = data.maxPropertyValue;
    maxBorrow.textContent = data.improvedPropertyValue;
    submitForm();
    return;
  }
  if (currentStep < totalSteps) {
    stepElements[currentStep - 1].classList.remove('active');
    currentStep++;
    stepElements[currentStep - 1].classList.add('active');
    updateButton();
    disableMainButton();
  }
}

Немає коду, який перевіряє answers.looking (або значення radio елемента name="looking") для розгалуження потоку.

Граничні випадки та пастки для розробників

Наступні проблеми виявлені при інспекції живого JS. Вони зламають вашу реалізацію, якщо їх не врахувати.

1. disableMainButton() жорстко кодує номери кроків

Логіка перевіряє конкретні номери кроків: крок 2 = maritalStatus, крок 3 = income/deposit, крок 4 = name/email/phone. Для нових потоків (рефінансування, дослідник) ці номери не збігатимуться. Потрібно рефакторити на перевірку полів поточного кроку, а не жорстко закодованих номерів.

2. totalSteps = 5 жорстко закодовано

Контролює індикатор прогресу та логіку кнопки "Submit". Рефінансування має 6 кроків. Дослідник має 2–4. Потрібно зробити динамічним для кожного потоку.

3. submitToContactForm7() вже відправляє поля рефінансування — як прочерки

r.append("mortgage-rate", "-");
r.append("still-owe", "-");
r.append("loan-term", "-");
r.append("important", "-");

Це важливо: CF7 форма #72896 вже має поля для даних рефінансування — вони просто жорстко закодовані як "-". Для потоку рефінансування заповніть їх реальними відповідями замість додавання нових полів. Маппінг HubSpot може вже бути налаштований.

4. updateButton() використовує totalSteps для Submit vs Continue

З динамічною кількістю кроків потрібно перевіряти загальну кількість поточного потоку, а не глобальний totalSteps.

5. answers та formAnswers — окремі об'єкти

answers живить калькулятор. formAnswers відправляється. Для рефінансування додайте дані до відправки, маппячи на існуючі назви полів CF7: mortgage-rate, still-owe, loan-term, important.

6. Event listeners прив'язані до ID елементів

Існуючий код прив'язує addEventListener до #income, #deposit, #state тощо по ID. Нові поля рефінансування потребують власних listeners для оновлення об'єкту answers та виклику disableMainButton().

7. Відсутня аналітика

Квіз не має жодних подій GTM, Google Analytics або Facebook Pixel. Немає трекінгу якого шляху йде користувач, де відвалюється, або конверсій по потоку. Розгляньте додавання dataLayer.push при реалізації розгалуження.

8. Немає OTP/верифікації телефону

Незважаючи на згадки OTP у деяких потоках, квіз на головній сторінці не має SMS верифікації. Телефон — звичайне текстове поле з валідацією формату AU на стороні клієнта.

9. stepElements виконується один раз при завантаженні

document.querySelectorAll(".step") захоплює всі елементи .step при завантаженні сторінки. Якщо ви додаєте нові div кроків динамічно — вони не будуть у цьому NodeList. Додайте весь HTML кроків при збірці, або перезапитайте після додавання.

10. Індикатор прогресу "/ 5" жорстко закодований у HTML

Індикатор прогресу:

<div class="progress_indicator">
  <span id="current_step">1</span> / 5
</div>

JS оновлює текст #current_step, але / 5 — статичний HTML. Щоб зробити динамічним, загорніть загальну кількість у <span>, щоб JS міг змінити її для кожного потоку.

11. Кроки мають різну висоту контенту — без анімації

Висоти кроків: Крок 1 = 376px, Крок 2 = 531px, Крок 3 = 492px. Картка квізу не має фіксованої висоти, переходів чи анімацій. При зміні кроків висота стрибає миттєво. Розгляньте додавання transition: height 0.3s або min-height.

12. Кроки використовують display: none для приховування

Приховані кроки мають display: none. JS видаляє клас .active з поточного кроку і додає до наступного. Переконайтесь, що ваш CSS прив'язує .active до display: block, а стандартний .step має display: none. Нові div кроків потребують тих самих CSS правил.

4. Шлях 1 — Купівля нерухомості

Зберегти наявні кроки — виправити лише розгалуження

Зберегти калькулятор позики як хук для конверсії.

Кроки ситуації та фінансів (подружжя/одинак, першовласник, штат, утримавці, дохід, депозит) не передаються на backend — Typeform збирає їх все заново. Однак вони живлять калькулятор позики, який показує результат "$550k → $750k з нашою допомогою". Це сильний психологічний хук, що керує конверсіями, тому ми його зберігаємо.

Єдина зміна, необхідна для купівлі, це переконатися, що логіка розгалуження маршрутизує сюди правильно — наявний 5-кроковий потік повністю підходить.

Чому зберегти додаткові кроки?

Екран результату калькулятора позики ("Ви можете позичити $550,000 → $750,000 з нашою допомогою!") є перевіреним хуком конверсії. Щоб показати це число, квіз потребує введення доходу та депозиту для живлення calculateLoanCapacity(). Дані не передаються на backend, але вони живлять калькулятор, який керує конверсіями.

Показує розраховану кредитоспроможність за допомогою calculateLoanCapacity(answers), а потім up-sell "з нашою допомогою". Це хук, що керує конверсіями.

Не потрібні зміни до самого потоку купівлі

Наявний 5-кроковий потік купівлі працює правильно. Єдине виправлення — переконатися, що nextStep() правильно маршрутизує сюди (що він робить за замовчуванням, оскільки це наявний лінійний потік). Дані ситуації/фінансів живлять калькулятор на стороні клієнта — їх не потрібно передавати.

5. Шлях 2 — Будівництво (як Купівля)

Зберегти — той самий потік, що й Купівля

Будівництво слідує тому самому шляху, що й Купівля.

Коли користувач обирає "Будівництво", вони проходять той самий 5-кроковий потік, що й Купівля: домогосподарство → дохід та заощадження → контактні дані → результат кредитоспроможності. Значення looking зберігається як "build" у передачі форми, щоб команда знала, що це запит на будівництво.

6. Шлях 3 — Комерційний / бізнес-кредит (тупик)

Обмеження пропускної здатності — м'яка розвантаження

Комерційні/бізнес-кредити пропонуються, але HG наразі на межі пропускної здатності.

Однакова схема як у будівництві — показати повідомлення, що пояснює обмежену пропускну здатність через складність, а потім перенаправити на інші варіанти.

Комерційні кредити — наразі на межі пропускної здатності

Чому два форки?

Опціональна перевірка кредитоспроможності — ключовий інсайт: нав'язування фінансових даних тому, хто сказав "просто дивлюсь", занадто агресивне і спричинить відтік. Але пропозиція як обмін цінністю ("дізнайтесь скільки ви можете позичити за 30 секунд") конвертує цікавих дослідників у теплі ліди з повними даними — без тиску контактних даних чи перевірки телефону.

Перенаправлення на основі етапу захоплюють користувачів, які обрали "дивлюсь", але насправді далі на шляху. Замість того, щоб давати їм розмитий досвід, направте їх у повні потоки Купівлі чи Рефінансування, де вони отримають найбільше цінності.

Передача даних

Дослідники, які обирають перевірку кредитоспроможності, збирають ті ж поля, що й кроки ситуації + фінансів Купівлі (maritalStatus, isFirstOwner, state, dependants, income, deposit). Ці дані подаються безпосередньо в існуючу функцію calculateLoanCapacity(). Передача форми включає looking=exploring та exploreStage, щоб CRM могла відрізнити ці ліди від прямих запитів на купівлю.

9. Нотатки щодо реалізації

Основна зміна: розгалуження у nextStep()

Основне виправлення полягає у зміні nextStep() для читання answers.looking після кроку 1 і маршрутизації на правильний потік. Ось запропонований псевдокод:

// Запропонована рефакторна nextStep()
function nextStep() {
  const looking = answers.looking;

  // Після кроку 1: розгалуження на основі вибору
  if (currentStep === 1) {
    switch (looking) {
      case 'purchase':
        goToStep('purchase_situation'); // наявний крок_2 (зберегти повний 5-кроковий потік з калькулятором)
        break;
      case 'build':
        goToStep('purchase_situation'); // Той самий потік, що й purchase
        break;
      case 'commercial':
        showDeadEndScreen('commercial');
        return;
      case 'refinance':
        goToStep('refi_situation');
        break;
      case 'exploring':
        goToStep('explore_stage'); // Крок 2: На якому етапі?
        break;
    }
    updateProgressIndicator();
    return;
  }

  // Дивлюсь крок 2: розгалуження на основі етапу
  if (looking === 'exploring' && currentStep === 'explore_stage') {
    const stage = answers.exploreStage;
    if (stage === 'ready_to_buy') {
      answers.looking = 'purchase';
      goToStep('purchase_situation');
      return;
    }
    if (stage === 'curious_refi') {
      answers.looking = 'refinance';
      goToStep('refi_situation');
      return;
    }
    // Ранній етап: показати форк кредитоспроможності
    goToStep('explore_borrowing_fork');
    return;
  }

  // Форк кредитоспроможності для дослідників
  if (looking === 'exploring' && currentStep === 'explore_borrowing_fork') {
    if (answers.wantsBorrowingPower) {
      goToStep('explore_situation'); // збір домогосподарства + фінансів
    } else {
      goToStep('explore_resources'); // перехід до ресурсів
    }
    return;
  }

  // У межах кожного потоку: перейти до наступного кроку в цьому потоку
  advanceCurrentFlow();
}

// Помічник: показати екран тупика для непідтримуваних послуг
function showDeadEndScreen(type) {
  hideAllSteps();
  const screen = document.querySelector(`.dead-end-${type}`);
  screen.classList.add('active');
  // Оновити індикатор прогресу на стан "завершено"
  updateProgressIndicator('dead-end');
}

// CTA тупика: перенаправити на потік купівлі
function redirectToPurchase() {
  answers.looking = 'purchase';
  goToStep('purchase_situation');
}

C: Повний переписування компонента
Перейти на правильну машину стану або React-подібний компонент		 Найбільш надійний. Тестовий. Правильне управління станом. Найбільше зусиль. Може бути недоправданим для 5-крокового квізу.

Рекомендація: Підхід B (масиви кроків на основі потоку)
