Lead Monitoring Proposal

Two options to ensure form/integration failures are detected within hours, not months. For discussion with Jayden — 7 April 2026.

Why This Is Needed

The CRS form corruption (Oct 2025 – Apr 2026) ran undetected for 6 months because no automated monitoring existed between form submission and HubSpot data arrival. The frontend showed "Thank you!" while data arrived corrupted. The existing Playwright test suite (142 tests) catches frontend bugs but cannot verify what arrives in HubSpot.

Two separate detection layers are proposed. They complement each other — Option A catches problems passively, Option B catches them proactively.

Option A: HubSpot Active Lists + Workflow Alerts

Recommended — do this first

Effort

30 minutes to set up. No code. No API keys. No external tools.

Detection Speed

Within minutes of a corrupted contact being created. Runs continuously — 24/7.

What to Build

Three active lists in HubSpot that auto-populate when matching contacts appear, plus a workflow that sends an alert email when a contact enters any of these lists.

List NameFilter CriteriaWhat It Catches
ALERT: Email as Firstname firstname CONTAINS .com
OR firstname CONTAINS .org
OR firstname CONTAINS @		 The exact bug that corrupted 90 contacts. If any form ever maps email → firstname again, this fires immediately.
ALERT: Quiz Data = Dashes what_is_your_current_mortgage_interest_rate_ IS EQUAL TO -
AND createdate IS AFTER 2026-04-03		 Hardcoded dash values. Date filter ensures it only catches NEW contacts (not the existing 167).
ALERT: No Form Attribution first_conversion_event_name IS UNKNOWN
AND createdate IS IN THE LAST 7 days		 Contacts entering HubSpot with no record of which form captured them. Rolling 7-day window.

Setup Steps

1
Create the active lists. HubSpot → Contacts → Lists → Create list → Active list. Add the filter criteria above. Save each list.
2
Create a workflow for each list. HubSpot → Automation → Workflows → Create workflow → Contact-based. Enrollment trigger: "Contact is a member of list [ALERT list]." Action: Send internal notification email to simon.heikkila@gmail.com + jayden.vecchio@huntergalloway.com.au.
3
Test it. Submit a test form with an email address in the name field. Verify the alert fires. Delete the test contact.

Pros & Cons

  • Pro: Zero code, zero maintenance, zero cost (uses existing HubSpot)
  • Pro: Fires within minutes — real-time detection
  • Pro: Works regardless of who deploys forms (CRS, future devs, WordPress plugin updates)
  • Con: Only catches known patterns (email-as-name, dashes, missing attribution). A novel bug type would be missed.
  • Con: Passive — relies on real leads being submitted. If a form breaks completely (zero submissions), this won't alert.

Option B: Automated E2E Form Verification (Playwright + HubSpot API)

Comprehensive — build after Option A

Effort

2–4 hours to build. Requires HubSpot Private App token. Runs as a cron job or CI schedule.

Detection Speed

Depends on schedule — daily catches issues within 24 hours. Weekly within 7 days.

How It Works

1
Playwright submits a test form on the live site with known sentinel values:
firstname: __monitor_test
email: monitor.[timestamp]@heikkila.dev
phone: 0400000000
2
Wait 90 seconds for HubSpot ingestion (CF7 → HubSpot Collected Forms pipeline).
3
Query HubSpot API to find the test contact by email. Verify:
  • firstname = __monitor_test (not an email address)
  • email = the submitted email
  • Quiz fields (if applicable) contain real values, not "-"
  • transaction_type is populated (on forms that should send it)
4
Delete the test contact from HubSpot via API to keep the CRM clean.
5
Report results. If any check fails → send alert email. If all pass → log success silently.

Example Test Code

// tests/monitoring/e2e-hubspot-verification.spec.ts
import { test, expect } from '@playwright/test';

const HUBSPOT_TOKEN = process.env.HUBSPOT_PRIVATE_APP_TOKEN;
const SENTINEL_NAME = '__monitor_test';

test('Homepage quiz → HubSpot field mapping is correct', async ({ page }) => {
  const timestamp = Date.now();
  const testEmail = `monitor.${timestamp}@heikkila.dev`;

  // Step 1: Submit the homepage quiz
  await page.goto('https://www.huntergalloway.com.au/');
  // ... fill quiz steps, submit with sentinel values ...

  // Step 2: Wait for HubSpot ingestion
  await page.waitForTimeout(90_000);

  // Step 3: Query HubSpot API
  const res = await fetch('https://api.hubapi.com/crm/v3/objects/contacts/search', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${HUBSPOT_TOKEN}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      filterGroups: [{
        filters: [{ propertyName: 'email', operator: 'EQ', value: testEmail }]
      }],
      properties: ['firstname', 'email', 'what_is_your_current_mortgage_interest_rate_',
                    'how_much_do_you_still_owe_on_your_mortgage_',
                    'what_is_your_remaining_loan_term_',
                    'what_s_most_important_to_you_in_refinancing_']
    })
  });
  const data = await res.json();

  // Step 4: Verify field mappings
  expect(data.total).toBe(1);
  const contact = data.results[0].properties;
  expect(contact.firstname).toBe(SENTINEL_NAME);        // Not an email
  expect(contact.firstname).not.toContain('@');          // Double check
  expect(contact.firstname).not.toContain('.com');       // Triple check
  // Quiz fields should not be dashes (if refinance path was taken)

  // Step 5: Cleanup — delete test contact
  const contactId = data.results[0].id;
  await fetch(`https://api.hubapi.com/crm/v3/objects/contacts/${contactId}`, {
    method: 'DELETE',
    headers: { 'Authorization': `Bearer ${HUBSPOT_TOKEN}` }
  });
});

Prerequisites

  • HubSpot Private App token with crm.objects.contacts.read and crm.objects.contacts.write scopes
  • An email domain for test submissions (e.g. @heikkila.dev) that won't pollute real lead data
  • Playwright installed (already in the repo)

Pros & Cons

  • Pro: Catches ANY form breakage — not just known patterns. Tests the full pipeline end-to-end.
  • Pro: Catches total form failure (zero submissions) — Option A can't do this.
  • Pro: Catches new bug types that haven't been seen before.
  • Pro: Provides ongoing regression testing as forms change.
  • Con: Creates test contacts in HubSpot (cleaned up automatically, but briefly visible).
  • Con: Requires HubSpot Private App setup (one-time, 10 minutes).
  • Con: 90-second wait per form makes the test suite slower (~6 min for all 4 forms).

Recommendation

This Week

Set up Option A (HubSpot lists + alerts). Takes 30 minutes. Catches the known failure modes immediately. Zero risk.

When React Ships

Build Option B (E2E verification) as part of the React site launch QA. Run it daily via GitHub Actions. This becomes the permanent safety net.

Note on Monday.com integration: If the sales team processes leads in Monday.com rather than directly in HubSpot, Option B can be extended to also verify arrival in Monday.com via their API. This requires Monday.com API access (pending from Jayden).
