What Can Break

Single points of failure and known risks across the entire lead system, ranked by impact. These are the things to monitor and test regularly. Updated 4 April 2026 with two new critical risks from the Zapier webhook / API key incident.

Critical — Leads Lost Silently

Critical HubSpot tracking code goes down
If the HubSpot JavaScript tracking code fails to load (CDN issue, script blocked, WordPress theme update breaks it), every form submission still succeeds in WordPress but never reaches HubSpot. The visitor sees a success message. The lead is silently lost. There is no fallback, no error alert, no WordPress-side backup. This is the single biggest risk in the system.
Affects: All web forms (F-01 through F-11, excluding F-12 call centre)  ·  Detection: Monitor HubSpot form submission count daily. A sudden drop to zero = tracking code is broken.
Debug steps:
1. Open any page → DevTools → Console → search for hs-script-loader or HubSpot errors
2. Check HubSpot dashboard for recent form submission count — if zero, tracking is down
3. In WP Admin → check theme header/footer for the HubSpot embed script (js.hs-scripts.com/4607234.js)
4. Test: submit a form, wait 2 min, search for the email in HubSpot Contacts
Critical HS-WF-008 (New Lead SMS) is turned off or breaks
This single workflow handles the SMS alert that tells brokers about new web leads AND assigns the contact owner. If it's off, leads still reach HubSpot but nobody gets notified and no owner is assigned. Leads sit uncontacted.
Affects: All web form leads  ·  Detection: If brokers stop receiving SMS alerts for new leads, check this workflow first.
Debug steps:
1. Go to HubSpot Workflows → search "WF-008" or "New Lead SMS" → check it's ON
2. Check workflow logs for recent errors (enrollment failures, SMS send errors)
3. Verify the SMS provider integration is still active (check connected apps)
4. Test: create a test contact manually, check if SMS fires within 5 min
Critical HS-WF-002/003 (Broker Assignment) breaks
These workflows map the adviser property to the correct broker. If they break, leads get assigned to no one or the wrong broker. The MANUAL version (HS-WF-003) has enrolled 221,000+ contacts — it's the most heavily used workflow in the system.
Affects: All leads  ·  Detection: Check if contact owner is blank after form submissions.
Debug steps:
1. HubSpot Contacts → filter by "Contact owner is unknown" + created today → if many, assignment is broken
2. Check WF-002 and WF-003 are ON and not paused in Workflows
3. Check the adviser property is being set correctly (it maps CA codes to brokers)
4. Manual fix: use WF-003 (MANUAL version) to re-run assignment on affected contacts
Critical Zapier webhook integrations lost when forms are modified or recreated
Incident: 1-4 April 2026. CRS created new contact forms but did not copy over existing Zapier webhook integrations. The Refinance quiz form was also missing its webhook. An estimated 80+ leads dating back to October 2025 were silently lost — forms submitted successfully but data never reached the downstream systems (HubSpot/Monday.com). This is a silent failure with no error visible to the user or the form creator.
Affects: Any form with a Zapier webhook integration  ·  Detection: Compare form submissions in WordPress vs contacts created in HubSpot. A mismatch = broken webhook.
Mandatory checklist — when creating or modifying any form:
1. Document all existing Zapier webhook URLs attached to the form BEFORE making changes
2. After creating/modifying the form, verify every Zapier webhook has been carried over
3. Test end-to-end: submit the form → check Zapier task history → confirm contact appears in HubSpot and Monday.com
4. Do NOT deploy until E2E verification is complete
5. Check the WP Contact Form HubSpot Pro plugin API key is valid (it can expire silently)
Critical WP Contact Form HubSpot Pro API key expires silently
Incident: April 2026. The API key for the WP Contact Form HubSpot Pro plugin expired, causing leads to stop flowing from WordPress to HubSpot. There is no alert or notification when this happens — leads are silently lost.
Affects: All CF7 forms that rely on this plugin for HubSpot integration  ·  Detection: If leads stop appearing in HubSpot but forms still submit in WordPress, check this API key first.
Debug steps:
1. WP Admin → Settings → WP Contact Form HubSpot Pro → check if API key is valid
2. Re-enter the API key if expired
3. Submit a test form and verify the contact appears in HubSpot within 2 minutes
4. Add API key expiry date to a calendar reminder for proactive renewal

High — Features Broken, Partial Data Loss

High JavaScript fails on mortgage quiz (F-03)
The mortgage calculator quiz is entirely JS-dependent. If JavaScript fails to load (script error, CDN issue, browser extension conflict), the quiz silently doesn't work. The calculator still shows results, but the lead capture form never appears. No error message is shown to the user.
Affects: F-03 Mortgage Quiz  ·  Detection: Playwright test T-025 checks quiz form loads, but can't catch all JS errors.
Debug steps:
1. Open /mortgage-calculator/ → DevTools → Console → look for JS errors
2. Complete the calculator → check if the contact form appears after the results
3. If form doesn't appear: check Network tab for failed script loads
High Phone formatting is JS-only (all forms)
Phone numbers are auto-formatted to Australian format (0412 345 678) by JavaScript. If JS breaks, unformatted numbers are accepted and submitted. HubSpot may reject or incorrectly parse these. No server-side phone validation has been confirmed.
Affects: F-01, F-02, F-04, F-07  ·  Detection: Check for malformed phone numbers in HubSpot contacts.
High Free Assessment custom dropdown breaks (F-02)
The transaction type dropdown on the Free Assessment form uses a custom .btn_select UI component, not a native HTML <select>. If the custom JS breaks, the dropdown won't open and users can't select a transaction type. Since it's required, the form becomes unusable.
Affects: F-02 Free Assessment (primary lead funnel)  ·  Detection: Playwright test T-015 checks dropdown options are present.
Medium Gravity Forms #9 and #10 still exist — destination unknown (F-05/F-06)
Correction (2 Apr 2026): Previously marked as "Resolved" but Playwright scan confirmed gform_wrapper_9 (YTD) and gform_wrapper_10 (LVR) are still present in the DOM. Both Gravity Forms are live on their respective calculator pages. TBD: Where do these forms submit to? Do they reach HubSpot, or do they go to a different destination (email, Zapier, etc.)?
Affects: F-05, F-06  ·  Detection: No Playwright tests currently cover the conditional broker form trigger.
High Homepage quiz has no branching — all paths follow identical flow (F-03)
The nextStep() function in the homepage quiz JavaScript does currentStep++ with no conditional branching. Regardless of selection (Purchase, Build, Commercial, Refinance, Exploring), users see the same 5 steps: situation → finances → contact → borrowing result. Refinance users are asked about deposit and first home buyer status. Build/Commercial users go through the full quiz instead of getting capacity screens.
Affects: F-03, F-09 (homepage quiz on / and /mortgage-calculator/)  ·  Detection: Playwright tests in forms/homepage-quiz-branching.spec.ts (4 bug regression tests)  ·  Fix spec: Quiz Flow Specification
High /lower-your-rate/ refinance quiz answers are silently discarded
The refinance quiz at /lower-your-rate/ collects 4 steps of quiz data (interest rate, loan amount, loan term, refinancing priority) but CF7 #14462 has no hidden fields to capture them. Only name/email/phone reach HubSpot. All quiz engagement data is lost on submit. This explains 12+ "empty" submissions in HubSpot with no transaction type from this page.
Affects: /lower-your-rate/ page  ·  Detection: Playwright tests in forms/lower-your-rate.spec.ts (3 bug regression tests)  ·  Fix spec: Form Data Loss Report
Medium Equity calculator hero banner is broken (C-02)
The /equity-calculator/ page has two calculators. The working one (below fold, #h_equity_calculator) works fine. The hero banner calculator at the top is completely broken: the input has no name/id and no JS reads it, the "Calculate" button is just <a href="/equity-calculator"> which reloads the page, and there's only 1 input where equity calculations need 2 (property value AND loan balance).
Affects: C-02 Equity Calculator  ·  Detection: Playwright tests in calculators/equity-calculator-banner.spec.ts (4 bug regression tests)
